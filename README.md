# claims-agent

A HIPAA-shaped, member-scoped agentic AI assistant that drafts plain-English claim explanations from synthetic health insurance data. Built on AWS Bedrock with Claude Haiku 4.5, LangChain.js, and a four-tool agent loop.

**All data in this system is synthetic.** No real PHI, no real claims, no affiliation with any health plan or insurer.

**Live demo:** https://claims-agent.rangbull-labs.com
**Design decisions:** [docs/DESIGN_DECISIONS.md](docs/DESIGN_DECISIONS.md)
**Lessons learned:** [docs/LESSONS_LEARNED.md](docs/LESSONS_LEARNED.md)

## What it does

A member asks a question in plain English. The agent classifies the inquiry, looks up that member's claims in DynamoDB (scoped by deterministic middleware — the LLM never sees unscoped data), retrieves relevant plan policy language from a Bedrock Knowledge Base, and drafts a plain-English response for human review. No tool in the system can send a message. The escalation guard intercepts legal, medical-advice, and self-harm language before the model is invoked. Everything runs against ~20 synthetic members, ~50 synthetic claims, and three synthetic plan documents generated by scripts in this repo.

## Architecture

The four-tool loop follows the framework from MIT Sloan's *Implementing Agentic AI* program (April 2026 cohort). Each step is auditable in the trace store independently.

- **Runtime:** TypeScript · Node.js 20 · AWS Lambda (Function URL)
- **Agent:** LangChain.js v1 (`createAgent` + LangGraph + `modelCallLimitMiddleware`)
- **Model:** Claude Haiku 4.5 via Bedrock inference profiles (Sonnet 4 for eval comparison)
- **Embeddings:** Amazon Titan v2
- **Knowledge base:** Bedrock KB with Pinecone vector store
- **State:** DynamoDB (members, claims, agent traces)
- **Frontend:** React 18 · Vite · TypeScript · Tailwind
- **Hosting:** Firebase Hosting · Cloudflare (custom subdomain + edge rate limiting)
- **Monorepo:** pnpm workspaces (`backend/` + `frontend/`)

## Architectural commitments

Load-bearing. Full reasoning in [DESIGN_DECISIONS.md](docs/DESIGN_DECISIONS.md).

- **Four-tool deterministic decomposition** — classify → lookup → retrieve → draft
- **Member scoping at the data layer** — `memberId` captured in tool closures before the LLM runs; tool input schemas don't accept a member parameter
- **Deterministic escalation guard** — runs before the agent, sub-150ms; legal, medical-advice, and self-harm signals routed by string matching, not by the model
- **Draft-only output** — no "send" tool exists; every response goes to a review queue
- **Full audit trail** — every tool call persisted to DynamoDB with inputs, outputs, confidence, and scope-violation flags
- **Scope visible at the UI layer** — chat messages tagged with the member context active at send time; member switches render explicit section dividers

## What this is NOT

Synthetic data only. No HIPAA compliance achieved — the architecture is *shaped like* a compliant system; it does not *achieve* compliance. Two safeguard layers are deferred: the deterministic policy engine (rule-based draft validation) and the external audit monitor. No multi-turn memory. No access control on the Function URL. Full gap list in [DESIGN_DECISIONS.md Section 8](docs/DESIGN_DECISIONS.md#8-what-this-is-not).

## Local development

```bash
pnpm install
cp .env.example .env                                # fill in AWS credentials + resource IDs
cp frontend/.env.local.example frontend/.env.local  # set VITE_LAMBDA_URL

pnpm --filter @claims-agent/backend generate-data   # create synthetic fixtures
pnpm --filter @claims-agent/backend seed-data       # load into DynamoDB
pnpm --filter @claims-agent/backend ingest-kb       # upload policy docs + trigger KB ingestion

pnpm --filter @claims-agent/frontend dev             # http://localhost:5173
```

Full AWS provisioning (DynamoDB tables, Lambda, Bedrock KB, IAM) is documented in [docs/AWS_SETUP.md](docs/AWS_SETUP.md).
