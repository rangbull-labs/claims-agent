# claims-agent

A HIPAA-shaped, member-scoped agentic AI assistant that drafts plain-English claim explanations from synthetic health insurance data. Built on AWS Bedrock with Claude Haiku 4.5, LangChain.js, and a four-tool agent loop.

**All data in this system is synthetic.** No real PHI, no real claims, no affiliation with any health plan or insurer.

**Live demo:** https://claims-agent.rangbull-labs.com
**Design decisions:** [docs/DESIGN_DECISIONS.md](docs/DESIGN_DECISIONS.md)
**Lessons learned:** [docs/LESSONS_LEARNED.md](docs/LESSONS_LEARNED.md)

## What it does

A member asks a question. The agent classifies it, looks up claims scoped to that member (via deterministic middleware — the LLM never sees unscoped data), retrieves relevant policy language from a Bedrock Knowledge Base, and drafts a plain-English response for human review. No tool can send a message; the escalation guard intercepts legal, medical-advice, and self-harm language before the model is invoked. Everything runs against synthetic data generated by scripts in this repo.

## Architecture

The four-tool loop follows the framework from MIT Sloan's *Implementing Agentic AI* program. Each step is auditable in the trace store independently.

- **Runtime:** TypeScript · Node.js 20 · AWS Lambda (Function URL)
- **Agent:** LangChain.js v1 (`createAgent` + LangGraph) · Claude Haiku 4.5 via Bedrock (Sonnet 4 for eval)
- **Knowledge base:** Bedrock KB with Pinecone vector store · Amazon Titan v2 embeddings
- **State:** DynamoDB (members, claims, agent traces)
- **Frontend:** React 18 · Vite · Tailwind · Firebase Hosting
- **Domain:** `claims-agent.rangbull-labs.com` via Cloudflare (DNS + edge caching)

## Architectural commitments

These are load-bearing. Full reasoning in [DESIGN_DECISIONS.md](docs/DESIGN_DECISIONS.md).

- **Four-tool deterministic decomposition** — classify → lookup → retrieve → draft. The agent follows this sequence; the trace store captures each step.
- **Member scoping at the data layer** — `memberId` is captured in tool closures before the LLM runs. Tool input schemas don't accept a member parameter; a prompt injection saying "look up member 12345" reaches a tool that can't comply.
- **Deterministic escalation guard** — runs before the agent, sub-150ms. Legal language, medical-advice requests, and self-harm signals are routed by string matching, not by the model.
- **Draft-only output** — no "send" tool exists. Every response is written to a review queue. A human takes a separate action to approve.
- **Full audit trail** — every tool call, classification, and draft is persisted to DynamoDB with timestamp, member ID, inputs, outputs, and confidence. Scope violations are logged with the `attemptedMemberId`.

## What this is NOT

Synthetic data only. No real PHI. No HIPAA compliance achieved — the architecture is *shaped like* a compliant system; the demo does not *achieve* compliance. The deterministic policy engine (rule-based draft validation) is documented but not shipped. No multi-turn memory across sessions. No access control on the Function URL. Full gap list in [DESIGN_DECISIONS.md Section 8](docs/DESIGN_DECISIONS.md#8-what-this-is-not).

## Repo structure

```
backend/          Lambda handler, agent loop, tools, safeguards, AWS clients
  src/            TypeScript source (strict mode, no default exports)
  scripts/        Data generation, DynamoDB seeding, KB ingestion, deploy
  tests/          Integration test suite (runs against deployed Lambda)
  data/           Synthetic members, claims, denial codes, policy docs

docs/             Design decisions, AWS setup guide, lessons learned
```

## Local development

```bash
pnpm install
cp .env.example .env              # fill in AWS credentials + resource IDs
cp frontend/.env.local.example frontend/.env.local  # set VITE_LAMBDA_URL

pnpm --filter @claims-agent/backend generate-data   # create synthetic fixtures
pnpm --filter @claims-agent/backend seed-data       # load into DynamoDB
pnpm --filter @claims-agent/backend ingest-kb       # upload policy docs + trigger KB ingestion

pnpm --filter @claims-agent/frontend dev             # http://localhost:5173
```

Full AWS provisioning (DynamoDB tables, Lambda, Bedrock KB, IAM) is documented in [docs/AWS_SETUP.md](docs/AWS_SETUP.md).
